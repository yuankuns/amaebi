# Rust Development Rules for AI Agents

When writing Rust code in this project, adhere strictly to the following guidelines. These are optimized for safe, idiomatic, and maintainable Rust code generated by AI.

## 1. Toolchain & Ecosystem
- Use **Rust 2021 Edition**.
- Always use **`cargo fmt`** after writing or modifying code to ensure standard formatting.
- Always run **`cargo clippy -- -D warnings`** and fix all lints before considering a feature complete.
- Manage dependencies via `cargo add` or by cleanly editing `Cargo.toml`.

## 2. Error Handling
- Use the **`anyhow`** crate for application-level error handling (e.g., binaries, CLI, Daemon). Return `anyhow::Result<T>`.
- Use the `?` operator for error propagation instead of verbose `match` statements where possible.
- Add context to errors using `.context("...")` or `.with_context(|| "...")` from the `anyhow::Context` trait when an operation fails (e.g., `File::open(path).context("Failed to open config file")?`).
- For library crates (if extracted), use **`thiserror`** to define custom strongly-typed errors.

## 3. Async Programming
- Use **`tokio`** as the default async runtime.
- Prefer `#[tokio::main]` for the entry point.
- When dealing with I/O (like Unix Sockets or HTTP), use `tokio::net` and `tokio::io` traits (e.g., `AsyncReadExt`, `AsyncWriteExt`).

## 4. Ownership and Borrowing
- **Avoid arbitrary `.clone()` or `.to_string()`**: Do not just blindly clone to satisfy the borrow checker. If a reference `&str` or `&T` works, use it. Clone only when ownership is genuinely required (e.g., moving data into a spawned async task).
- When passing data to async tasks spawned via `tokio::spawn`, use `Arc` for shared state rather than deep cloning large structs.

## 5. Modularity & Project Structure
- Keep `main.rs` small. Move logic into `lib.rs` or separate modules (e.g., `daemon.rs`, `client.rs`, `cli.rs`).
- Use `clap` (with the `derive` feature) for CLI argument parsing. Define strict struct representations for your CLI args.

## 6. Safety & Unwrapping
- **Never use `.unwrap()` or `.expect()`** in production code unless you can absolutely guarantee it will never panic, and even then, add a comment explaining *why* it's safe. Use `?` and return proper errors instead.

## 7. Logging & Tracing
- Use the **`tracing`** crate along with `tracing-subscriber` for logging instead of `println!` (except when explicitly communicating with the CLI client via standard output).

Please memorize these rules and apply them to all code generated for `tmux-copilot`.
